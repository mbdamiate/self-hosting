## Context

Today, VM lifecycle logic lives in three independent bash scripts (2124 lines total) with zero automated tests — `TESTING.md` is a fully manual roteiro that requires a real KVM host, and stays that way regardless of implementation language. Duplication has already drifted rather than stayed identical: `WORK_DIR` is computed differently in `debian-vm-backup.sh` (via a separate `DEFAULT_DEST_ROOT`) than in `debian-vm-setup.sh`/`debian-vm-cleanup.sh`, and `confirm()` (cleanup, bypassed by `--vm-only`/`--purge-all`) vs `confirm_destructive()` (backup, never bypassed) implement the same intent inconsistently.

Existing specs under `openspec/specs/` (`vm-setup-rerun-recovery` in particular) phrase requirements in terms of specific `virsh` subcommands (`domiflist`, `dumpxml`, `domstate`, `domblklist`, ...), confirming this project's implicit rule: **libvirt is the single source of truth for anything it can report; nothing the tooling caches is trusted over a live query.** This design must preserve that rule, not just avoid breaking the specific behaviors it currently protects.

## Goals / Non-Goals

**Goals:**
- One Go binary (`vmctl`) with subcommands replacing `debian-vm-setup.sh`, `debian-vm-cleanup.sh`, `debian-vm-backup.sh`, sharing flag parsing, the `--name` → `WORK_DIR` convention, preflight checks, and confirmation logic.
- Preserve every existing requirement in `openspec/specs/` unchanged in observable behavior — same `virsh`/`virt-install`/`qemu-img`/`cloud-localds` invocations, same guarantees, same error messages pointing at the literal `virsh` command to inspect manually.
- Add `vmctl list`/`vmctl status`: a live, uncached, fleet-wide view.
- Add one consolidated metadata record for guest-only facts (admin sudo policy, log-forwarding, guest firewall policy) replacing the three scattered dotfiles.
- Make the shared plumbing (flag parsing, workdir resolution, preflight checks) unit-testable without a KVM host.

**Non-Goals:**
- Libvirt Go bindings (`go-libvirt`/`libvirtxml`) replacing `virsh` exec calls — deferred to a possible future phase ("Level 2").
- Any daemon, HTTP, or gRPC service — `vmctl` is a single process that runs and exits, exactly like the scripts it replaces ("Level 3", deferred/rejected for this project's scale).
- Backward-compatible shims for the old `./scripts/debian-vm-*.sh` invocation form — this is a clean cutover once `vmctl` covers equivalent behavior.
- Caching or persisting any fact libvirt already reports authoritatively (RAM, vCPUs, disk size, run state, network mode) — `vmctl list`/`status` always query live.

## Decisions

### Package layout
```
cmd/vmctl/main.go        -- entrypoint, subcommand dispatch
internal/cli/             -- shared flag/subcommand plumbing, --name/WORK_DIR resolution,
                              preflight checks (non-root, virsh present), Confirm(prompt, autoApprove)
internal/exec/             -- thin wrapper around exec.Command for virsh/virt-install/qemu-img/
                              cloud-localds/genisoimage/iptables; the single seam mocked in unit tests
internal/setup/            -- `vmctl setup`
internal/cleanup/          -- `vmctl cleanup`
internal/backup/           -- `vmctl backup snapshot|backup|restore|list`
internal/fleet/             -- `vmctl list` / `vmctl status`
internal/metadata/          -- vm-tooling-metadata read/write
```
Each subcommand package depends on `internal/cli` and `internal/exec`, never on each other — mirroring the current scripts' independence while sharing the seams that actually duplicated.

### Flag/subcommand library: stdlib `flag`, not Cobra
Considered `spf13/cobra` (the library `kubectl`-style tools typically use) vs. the standard library's `flag` package with one `flag.NewFlagSet` per subcommand.

Decision: stdlib `flag`. The current scripts have exactly zero runtime dependencies beyond system packages the tooling installs itself (`virsh`, `qemu-img`, etc.); introducing Cobra's dependency tree for 3-4 subcommands and ~16 flags is disproportionate. `--help` text is generated by iterating the `FlagSet`'s registered flags (still driven by one declaration per flag, still avoiding the current hand-written-text-vs-parsing-logic drift risk) without adding an external dependency.

### `exec.Command` wrapper, not libvirt bindings (reaffirms proposal scope)
All `virsh`/`virt-install`/`qemu-img`/`cloud-localds`/`genisoimage`/`iptables` calls go through one `internal/exec` seam:
```go
type Runner interface {
    Run(ctx context.Context, name string, args ...string) ([]byte, error)
}
```
Production code uses a `Runner` backed by `os/exec`; unit tests inject a fake `Runner` to verify flag parsing, workdir resolution, and branching logic (e.g., "existing VM found" vs. "not found") without a KVM host. This is the concrete testability gain motivating the rewrite — it does not remove the need for the manual `TESTING.md` roteiro, which still validates real `virsh`/QEMU behavior; it adds a layer underneath that.

### `vmctl list`/`status`: live query, zero persistence
Implementation loops `virsh list --all --name`, then for each name calls the same introspection the current setup script already performs at rerun time (`domiflist` for network mode, `dumpxml` for watchdog/on_crash, `domifaddr`/`net-dhcp-leases` for IP) via `internal/exec`, and renders a table with `text/tabwriter`. No new state is written anywhere. This deliberately avoids reintroducing the class of bug `vm-setup-rerun-recovery` was created to fix (trusting recorded state libvirt could have diverged from).

### Consolidated metadata: libvirt domain `<metadata>`, with a JSON-file fallback
Two options considered for replacing `.admin-sudo-policy`, `.log-forwarding-configured`, `.guest-firewall-policy`:

- **(a) One JSON file** under `WORK_DIR` (e.g. `meta.json`), typed via a Go struct.
- **(b) libvirt's native domain `<metadata>` XML element**, set via `virsh metadata <vm> --uri <namespace> --set '<xml/>'` and read via `virsh metadata <vm> --uri <namespace>`.

Decision made during implementation: **(a), the JSON file.** (b) was the original preference — it ties the metadata's lifecycle directly to the domain's own lifecycle, matching how `vm-cleanup-scope` already reasons about what `--vm-only`/`--purge-all` remove, and fits this project's style of pointing users at a literal `virsh` command to inspect state by hand. But committing to (b) required the empirical spike below, and the environment this was implemented in has no working libvirtd connection at all (`virsh` fails with a socket permission error, no VM has ever been created), so the spike could not be run. Rather than guess at `virsh metadata`'s untested XML-escaping/size behavior, (a) was implemented directly: it's fully testable without a real host and still satisfies every requirement in `specs/vm-tooling-metadata/spec.md` (written in terms of observable behavior, not storage mechanism). (a)'s known drift risk — the file surviving an `undefine` done outside `vmctl`, or `WORK_DIR` being deleted while the domain still exists — is accepted for now. **(b) remains the preferred direction**; revisit it once a real libvirtd host is available to spike against (see Open Questions).

### Rewrite order: smallest script first
Port `cleanup` (408 lines, least state) → `backup` (399 lines) → `setup` (1317 lines, most rerun/introspection logic) last, once the shared plumbing (`internal/cli`, `internal/exec`) has been proven against two smaller subcommands. Each old bash script is deleted only after its Go replacement passes the equivalent section of `TESTING.md` by hand.

## Risks / Trade-offs

- **[Risk]** Rewriting 2124 lines of bash that encode fixes from 22 archived changes risks silently dropping an edge case (e.g., a specific ordering guarantee in cleanup's network-then-VM-then-storage sequence). → **Mitigation**: port one script at a time, run the corresponding `TESTING.md` steps against the new binary before deleting the old script; keep both callable side by side until parity is confirmed.
- **[Risk]** The JSON-file metadata record (option (a), actually implemented) can drift from the domain's real lifecycle — surviving an `undefine` done outside `vmctl`, or missing if `WORK_DIR` is reconstructed some other way. → **Mitigation**: accepted for now since it can't diverge from anything libvirt itself claims (only guest-only facts live here); revisit moving to libvirt domain `<metadata>` (option (b)) once a real libvirtd host is available to spike against.
- **[Risk]** Go toolchain becomes a new build-time dependency not present today. → **Mitigation**: build-time only, not runtime; document as a prerequisite alongside the existing apt-based ones in `README.md`.
- **[Trade-off]** Losing bash's "the script *is* the documentation of what `virsh` command it runs" transparency is possible if error messages aren't deliberately carried over. → Treated as a hard requirement of the `vmctl-cli` capability spec, not an incidental nice-to-have.

## Migration Plan

1. Scaffold the Go module and package skeleton (`cmd/vmctl`, `internal/cli`, `internal/exec`).
2. Implement and unit-test the shared plumbing (flags, `WORK_DIR` resolution, preflight checks, `Confirm`).
3. Port `cleanup`, validate against `TESTING.md`, delete `debian-vm-cleanup.sh`.
4. Port `backup` (snapshot/backup/restore/list), validate, delete `debian-vm-backup.sh`.
5. Port `setup` (including all rerun-recovery introspection), validate, delete `debian-vm-setup.sh`.
6. Add `vmctl list`/`status`.
7. Implement `internal/metadata` (JSON-file option, since the `virsh metadata` spike couldn't be run — see Open Questions).
8. Update `README.md` and `TESTING.md` to the `vmctl` invocation form.

Rollback: every step is a separate commit; reverting to bash at any point is a normal git revert, since old scripts are only deleted after their replacement is validated, and no external system depends on the current bash entry points.

## Open Questions

- Does `virsh metadata` hold up under a real spike (size, escaping, round-trip fidelity) for the three guest-only facts it needs to carry? Couldn't be run in the implementation environment (no working libvirtd connection); still open, and the metadata implementation currently uses the JSON-file fallback (option (a)) as a result.
- Should `vmctl backup list` gain a `--all` flag to list backups across every VM once fleet-wide listing exists, or stay per-VM? Not required for this change but worth deciding before or during `vm-fleet-status` implementation.
- Does `on_crash=restart` actually trigger from `kill -9` on the QEMU process, or only from a libvirt-recognized guest-internal crash (e.g. via a `pvpanic` device)? Real-host testing (2026-07-20, `libvirtd.service` legacy monolithic daemon) found a VM created with default `on_crash=restart` (config confirmed correct and identical to what bash would generate) stayed `shut off` indefinitely after its QEMU process was killed with `SIGKILL` — no restart attempt logged anywhere (`journalctl -u libvirtd`, `systemd-machined`, or elsewhere). Since `TESTING.md`'s original roteiro was never run against a real host either, it's unclear whether `kill -9` was ever the right way to test this feature, or whether `on_crash=restart` itself doesn't work as documented on this libvirt/QEMU combination. Not treated as a vmctl-vs-bash regression (the generated domain XML is provably identical either way) — see `TESTING-vmctl.md` section 7 for the full finding and a suggested follow-up (test via an actual guest-panic mechanism instead of killing the host process).
- Why does a VM's disk file (`<name>.qcow2`) end up owned by `root:root` after a `virsh blockcommit --active --pivot` (used by `snapshot-delete` and the live `backup` path)? Real-host testing (2026-07-20) found this ownership drift causes a subsequent `backup-restore`'s unprivileged `qemu-img convert` to fail with "Permission denied". `virsh blockcommit` is invoked identically in bash, so this isn't a vmctl-vs-bash regression — it looks like `libvirtd` (running as root) takes ownership of the file while finalizing the pivot. Deliberately not "fixed" by having `vmctl` re-chown the disk after every blockcommit: the exact mechanism isn't understood yet, and blockcommit is called from three places (`cmdSnapshotDelete`, `cmdBackup`'s live path, and indirectly during retention) — a defensive chown added without understanding why the drift happens risks papering over a real problem instead of fixing it. See `TESTING-vmctl.md`'s backup-restore section for the workaround (`sudo chown` before restoring).
