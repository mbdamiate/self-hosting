## Why

The cloud-init `user-data` generated by `debian-vm-setup.sh` sets `package_upgrade: false` (debian-vm-setup.sh:488) and installs nothing to keep the VM patched afterward. That's a reasonable choice for a disposable local test VM created and torn down the same day, but a VM meant to run as a real server accumulates unpatched security advisories indefinitely with no mechanism to catch up on its own. Debian ships `unattended-upgrades` specifically to close this gap without requiring the operator to log in and run `apt` on a schedule.

## What Changes

- Install and configure `unattended-upgrades` via cloud-init so freshly created VMs apply Debian security updates automatically and on an ongoing basis, without requiring any operator action after boot.
- Restrict the automatic updates to the security origin only (not the full stable-updates/point-release stream), keeping behavior predictable and avoiding surprise package churn.
- Leave `package_upgrade: false` untouched — this is about ongoing patching after boot, not the one-time full-upgrade cloud-init already opts out of.
- Do not enable automatic reboot after kernel/library updates that require one; the VM keeps running on the old kernel until the operator reboots it manually.
- Add an opt-out flag, `--no-auto-updates`, for operators who need a pinned, unattended-upgrades-free environment (e.g., reproducible testing against a specific package snapshot). Default (flag omitted) is automatic security updates **on**.
- On fresh VM creation, print a one-time note in the setup output explaining that security updates apply automatically but reboots don't happen automatically.
- `--help` documents the new default and the opt-out flag. Per the existing `repository-readme` spec (README SHALL NOT restate the flag reference), `README.md` itself needs no new prose.

## Capabilities

### New Capabilities
- `vm-unattended-upgrades`: governs whether and how the VM applies Debian security updates automatically after boot, and how that's surfaced to the operator.

### Modified Capabilities
(none — orthogonal, additive behavior; does not change requirements of any existing spec)

## Impact

- `scripts/debian-vm-setup.sh`: argument parsing (`--no-auto-updates`), cloud-init `user-data` generation (`packages`, `write_files` for the unattended-upgrades config), `--help` text, one-time setup note.
- No impact on `README.md` — covered by the existing `--help`/`openspec/specs/` pointers per `repository-readme`.
- No impact on `scripts/debian-vm-cleanup.sh` (no host-side artifacts to clean up; this is guest-only, applied via cloud-init at first boot).
