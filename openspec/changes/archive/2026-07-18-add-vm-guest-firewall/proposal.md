## Why

The cloud-init `user-data` generated by `debian-vm-setup.sh` installs nothing to filter inbound traffic or react to brute-force attempts inside the VM itself. Today that's mitigated by the network mode being narrow by default (plain NAT is host-only, `--forward`/`--bridge` are opt-in), but a VM meant to run as a real server is expected to eventually have SSH — and possibly other services — reachable beyond the host. Without an in-guest firewall and a brute-force ban mechanism, the only thing standing between an attacker and the SSH daemon is the SSH key itself, with no second layer and no automatic response to repeated login attempts.

## What Changes

- Install and enable `ufw` inside the VM via cloud-init, default-denying inbound traffic except SSH (tcp/22), which stays allowed unconditionally so the operator is never locked out.
- Add an `--allow-port=PORT[,PORT...]` flag so operators can open additional guest-side TCP ports for their own services (e.g., a web app) at VM creation time.
- When `--forward=HOST_PORT:VM_PORT,...` is used, automatically add the VM-side ports to the guest firewall's allow list — traffic the operator already chose to forward from the host shouldn't then be silently dropped by the guest's own firewall.
- Install and enable `fail2ban` with its `sshd` jail active, banning IPs after repeated failed SSH login attempts using fail2ban's standard defaults. This is always on — it only reacts to failed auth, so it has no workflow cost even for local/disposable VMs.
- Add a `--no-guest-firewall` opt-out flag that skips installing/enabling `ufw` (and the port-related flags become no-ops), for operators who explicitly want the guest's network stack left wide open. `fail2ban` is not covered by this flag, since it has no equivalent downside.
- On fresh VM creation, print a one-time note listing which guest-side ports are allowed through the firewall.
- Record whether the guest firewall was enabled at creation in a small host-side marker file, so that `--forward` reapplied later against an already-existing VM (an officially supported flow per `vm-port-forward-reapplication`) can warn that the newly forwarded port also needs a manual `ufw allow` inside the guest — cloud-init can't reapply the guest firewall's own allow list after first boot.
- `--help` documents the new flags and defaults. Per the existing `repository-readme` spec (README SHALL NOT restate the flag reference), `README.md` itself needs no new prose.

## Capabilities

### New Capabilities
- `vm-guest-firewall`: governs the in-guest `ufw` firewall — default-deny inbound policy, the always-allowed SSH port, `--allow-port`, automatic allow-listing of `--forward` VM-side ports, and the `--no-guest-firewall` opt-out.
- `vm-guest-fail2ban`: governs the in-guest `fail2ban` SSH brute-force protection — always enabled, independent of the firewall opt-out.

### Modified Capabilities
(none — additive; does not change requirements of any existing spec)

## Impact

- `scripts/debian-vm-setup.sh`: argument parsing (`--allow-port`, `--no-guest-firewall`), cloud-init `user-data` generation (`packages`, `runcmd` for `ufw`/`fail2ban` setup), deriving VM-side ports from `--forward` for the firewall allow list, `--help` text, one-time setup note, the guest-firewall-state marker file and its `--forward`-reapplication warning.
- No impact on `README.md` — covered by the existing `--help`/`openspec/specs/` pointers per `repository-readme`.
- No impact on `scripts/debian-vm-cleanup.sh` (guest-only, applied via cloud-init at first boot, no host-side artifacts).
